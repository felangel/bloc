Les conventions de nommage suivantes sont simplement des recommandations et sont
complètement optionnelles. N'hésitez pas à utiliser les conventions de nommage
que vous préférez. Vous pouvez trouver certains exemples/documentation qui ne
suivent pas les conventions de nommage principalement pour des raisons de
simplicité/concision. Ces conventions sont fortement recommandées pour les
grands projets avec plusieurs développeurs.

## Conventions pour les Événements

Les événements doivent être nommés au **passé** car les événements sont des
choses qui se sont déjà produites du point de vue du bloc.

### Anatomie

`BlocSubject` + `Nom (optionnel)` + `Verbe (événement)`

Les événements de chargement initial doivent suivre la convention : `BlocSubject` +
`Started`

:::note

La classe de base de l'événement doit être nommée : `BlocSubject` + `Event`.

:::

## Conventions pour les États

Les états doivent être des noms car un état est simplement un instantané à un
moment particulier dans le temps. Il existe deux façons courantes de représenter
l'état : en utilisant des sous-classes ou en utilisant une seule classe.

### Anatomie

#### Sous-classes

`BlocSubject` + `Verbe (action)` + `State`

Lorsque l'état est représenté sous forme de plusieurs sous-classes, `State` doit
être l'un des suivants :

`Initial` | `Success` | `Failure` | `InProgress`

:::note

Les états initiaux doivent suivre la convention : `BlocSubject` + `Initial`.

:::

#### Classe Unique

`BlocSubject` + `State`

Lorsque l'état est représenté sous forme d'une seule classe de base, un enum
appelé `BlocSubject` + `Status` doit être utilisé pour représenter l'état :

`initial` | `success` | `failure` | `loading`.

:::note

La classe de base de l'état doit toujours être nommée : `BlocSubject` + `State`.

:::

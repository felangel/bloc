## v11.0.0

### `package:hydrated_bloc`

#### ❗🧹 `HydratedCubit` içindeki `storage` parametresini adlandırılmış (named) parametreye çevirin

:::note[Ne Değişti?]

hydrated_bloc v11.0.0'da, `HydratedCubit`'in isteğe bağlı `storage` parametresi
konumsal (positional) bir parametre yerine adlandırılmış (named) bir
parametreye dönüştürüldü.

:::

##### Gerekçe

Bu, ilk uygulama sırasında gözden kaçmıştı; yukarıdaki kırılma
değişikliği API'nin `HydratedBloc` ve `HydratedCubit` arasında tutarlı olmasını
sağlamak için yapıldı. Daha fazla bilgi için
[#4537](https://github.com/felangel/bloc/pull/4537) numaralı PR'a bakın.

**v10.x.x**

```dart
class CounterCubit extends HydratedCubit<int> {
  CounterCubit(Storage storage) : super(0, storage);
}
```

**v11.x.x**

```dart
class CounterCubit extends HydratedCubit<int> {
  CounterCubit(Storage storage) : super(0, storage: storage);
}
```

## v8.1.0

### `package:bloc`

#### ✨ `Bloc.observer` ve `Bloc.transformer` API'lerini Yeniden Sunma

:::note[Ne Değişti?]

bloc v8.1.0'da, `BlocOverrides`, `Bloc.observer` ve `Bloc.transformer` API'leri
lehine kullanımdan kaldırıldı.

:::

##### Gerekçe

`BlocOverrides` API'si, v8.0.0'da `BlocObserver`, `EventTransformer` ve
`HydratedStorage` gibi bloc'a özgü yapılandırmaların kapsamlandırılmasını
desteklemek amacıyla tanıtıldı. Saf Dart uygulamalarında değişiklikler iyi
çalıştı; ancak Flutter uygulamalarında yeni API çözdüğünden daha çok soruna
yol açtı.

`BlocOverrides` API'si, Flutter/Dart'taki benzer API'lerden ilham aldı:

- [HttpOverrides](https://api.flutter.dev/flutter/dart-io/HttpOverrides-class.html)
- [IOOverrides](https://api.flutter.dev/flutter/dart-io/IOOverrides-class.html)

**Sorunlar**

Bu değişikliklerin temel nedeni olmasa da, `BlocOverrides` API'si
geliştiriciler için ek karmaşıklık getirdi. Aynı etkiyi elde etmek için gereken
iç içe geçme miktarını ve kod satırlarını artırmasının yanı sıra,
`BlocOverrides` API'si geliştiricilerin Dart'taki
[Zone'lar](https://api.dart.dev/stable/2.17.6/dart-async/Zone-class.html)
hakkında sağlam bir anlayışa sahip olmasını gerektiriyordu. `Zone`'lar
yeni başlayanlara dost bir kavram değildir ve Zone'ların nasıl çalıştığını
anlamamak hatalara (başlatılmamış observer'lar, transformer'lar, storage
örnekleri gibi) yol açabiliyordu.

Örneğin, pek çok geliştirici şuna benzer bir kod yazıyordu:

```dart
void main() {
  WidgetsFlutterBinding.ensureInitialized();
  BlocOverrides.runZoned(...);
}
```

Yukarıdaki kod, zararsız görünmesine rağmen, izlenmesi zor pek çok hataya yol
açabilir. `WidgetsFlutterBinding.ensureInitialized` ilk olarak hangi zone
içinden çağrılırsa, `GestureBinding.initInstances` nedeniyle gesture
olaylarının (`onTap`, `onPressed` geri çağrıları gibi) işleneceği zone o
olur. Bu, `zoneValues` kullanmaktan kaynaklanan pek çok sorundan yalnızca
biridir.

Buna ek olarak Flutter, perde arkasında Zone'ları forklayan/manipüle eden pek
çok şey yapar (özellikle test çalıştırırken); bu da beklenmedik davranışlara
yol açabilir (ve birçok durumda geliştiricinin kontrolü dışında olan
davranışlara — aşağıdaki sorunlara bakın).

[runZoned](https://api.flutter.dev/flutter/dart-async/runZoned.html)
kullanımından dolayı, `BlocOverrides` API'sine geçiş Flutter'da (özellikle
Widget ve Entegrasyon Testleri etrafında) çeşitli hataların/sınırlamaların
keşfine yol açtı:

- https://github.com/flutter/flutter/issues/96939
- https://github.com/flutter/flutter/issues/94123
- https://github.com/flutter/flutter/issues/93676

Bu sorunlar, bloc kütüphanesini kullanan birçok geliştiriciyi etkiledi:

- https://github.com/felangel/bloc/issues/3394
- https://github.com/felangel/bloc/issues/3350
- https://github.com/felangel/bloc/issues/3319

**v8.0.x**

```dart
void main() {
  BlocOverrides.runZoned(
    () {
      // ...
    },
    blocObserver: CustomBlocObserver(),
    eventTransformer: customEventTransformer(),
  );
}
```

**v8.1.0**

```dart
void main() {
  Bloc.observer = CustomBlocObserver();
  Bloc.transformer = customEventTransformer();

  // ...
}
```

## v8.0.0

### `package:bloc`

#### ❗✨ Yeni `BlocOverrides` API'sini Tanıtma

:::note[Ne Değişti?]

bloc v8.0.0'da, `Bloc.observer` ve `Bloc.transformer` `BlocOverrides` API'si
lehine kaldırıldı.

:::

##### Gerekçe

Varsayılan `BlocObserver` ve `EventTransformer`'ı geçersiz kılmak için kullanılan
önceki API, hem `BlocObserver` hem de `EventTransformer` için bir global
singleton'a dayanıyordu.

Sonuç olarak şunlar mümkün değildi:

- Uygulamanın farklı bölümlerine kapsamlandırılmış birden fazla `BlocObserver`
  veya `EventTransformer` uygulamasına sahip olmak
- `BlocObserver` veya `EventTransformer` geçersiz kılmalarının bir paketle
  sınırlandırılması
  - Bir paket, `package:bloc`'a bağımlıysa ve kendi `BlocObserver`'ını
    kaydederse, paketin tüketicisi ya paketin `BlocObserver`'ının üstüne yazmak
    ya da paketin `BlocObserver`'ına raporlamak zorunda kalırdı.

Ayrıca, testler arasında paylaşılan global durum nedeniyle test etmek de daha
zordu.

Bloc v8.0.0, geliştiricilerin `BlocObserver` ve/veya `EventTransformer`'ı global
mutable bir singleton yerine belirli bir `Zone` için geçersiz kılmalarına
olanak tanıyan bir `BlocOverrides` sınıfı tanıtır.

**v7.x.x**

```dart
void main() {
  Bloc.observer = CustomBlocObserver();
  Bloc.transformer = customEventTransformer();

  // ...
}
```

**v8.0.0**

```dart
void main() {
  BlocOverrides.runZoned(
    () {
      // ...
    },
    blocObserver: CustomBlocObserver(),
    eventTransformer: customEventTransformer(),
  );
}
```

`Bloc` örnekleri, mevcut `Zone` için `BlocObserver` ve/veya
`EventTransformer`'ı `BlocOverrides.current` aracılığıyla kullanır. Eğer zone
için `BlocOverrides` yoksa, mevcut dahili varsayılanları kullanacaklardır
(davranış/işlevsellikte değişiklik olmaz).

Bu, her `Zone`'un kendi `BlocOverrides`'ı ile bağımsız çalışmasına imkân tanır.

```dart
BlocOverrides.runZoned(
  () {
    // BlocObserverA ve eventTransformerA
    final overrides = BlocOverrides.current;

    // Bu zone'daki bloc'lar BlocObserverA'ya raporlar
    // ve varsayılan transformer olarak eventTransformerA'yı kullanır.
    // ...

    // Daha sonra...
    BlocOverrides.runZoned(
      () {
        // BlocObserverB ve eventTransformerB
        final overrides = BlocOverrides.current;

        // Bu zone'daki bloc'lar BlocObserverB'ye raporlar
        // ve varsayılan transformer olarak eventTransformerB'yi kullanır.
        // ...
      },
      blocObserver: BlocObserverB(),
      eventTransformer: eventTransformerB(),
    );
  },
  blocObserver: BlocObserverA(),
  eventTransformer: eventTransformerA(),
);
```

#### ❗✨ Hata Yönetimi ve Raporlamayı Geliştirme

:::note[Ne Değişti?]

bloc v8.0.0'da `BlocUnhandledErrorException` kaldırıldı. Ek olarak,
yakalanmamış istisnalar her zaman `onError`'a raporlanır ve yeniden fırlatılır
(debug veya release modundan bağımsız olarak). `addError` API'si hataları
`onError`'a raporlar; ancak raporlanan hataları yakalanmamış istisna olarak
ele almaz.

:::

##### Gerekçe

Bu değişikliklerin amacı şudur:

- bloc işlevselliğini koruyarak dahili yakalanmamış istisnaları aşırı görünür
  kılmak
- kontrol akışını bozmadan `addError`'u desteklemek

Önceden, hata yönetimi ve raporlama uygulamanın debug ya da release modunda
çalışmasına bağlı olarak değişiyordu. Ek olarak, `addError` aracılığıyla
raporlanan hatalar debug modunda yakalanmamış istisnalar olarak ele alınıyordu;
bu da `addError` API'si kullanılırken (özellikle birim testleri yazarken) kötü
bir geliştirici deneyimine yol açıyordu.

v8.0.0'da, `addError` hataları raporlamak için güvenle kullanılabilir ve
`blocTest` hataların raporlandığını doğrulamak için kullanılabilir. Tüm hatalar
hâlâ `onError`'a raporlanır, ancak yalnızca yakalanmamış istisnalar yeniden
fırlatılır (debug veya release modundan bağımsız olarak).

#### ❗🧹 `BlocObserver`'ı abstract yapma

:::note[Ne Değişti?]

bloc v8.0.0'da, `BlocObserver` `abstract` bir sınıfa dönüştürüldü; bu,
`BlocObserver`'ın bir örneğinin doğrudan oluşturulamayacağı anlamına gelir.

:::

##### Gerekçe

`BlocObserver`'ın bir arayüz olması amaçlanmıştı. Varsayılan API uygulamaları
no-op olduğundan, sınıfın doğrudan örneklenmesi yerine genişletilmesi
amaçlandığını net bir şekilde iletmek için `BlocObserver` artık `abstract`
bir sınıftır.

**v7.x.x**

```dart
void main() {
  // Temel sınıfın bir örneğini oluşturmak mümkündü.
  final observer = BlocObserver();
}
```

**v8.0.0**

```dart
class MyBlocObserver extends BlocObserver {...}

void main() {
  // Temel sınıf örneklenemez.
  final observer = BlocObserver(); // HATA

  // Bunun yerine `BlocObserver`'ı genişletin.
  final observer = MyBlocObserver(); // TAMAM
}
```

#### ❗✨ Bloc kapalıysa `add` `StateError` fırlatır

:::note[Ne Değişti?]

bloc v8.0.0'da, kapalı bir bloc üzerinde `add` çağırmak `StateError`'a yol
açar.

:::

##### Gerekçe

Önceden, kapalı bir bloc üzerinde `add` çağırmak mümkündü ve dahili hata
yutuluyordu; bu da eklenen olayın neden işlenmediğini hata ayıklamayı
zorlaştırıyordu. Bu senaryoyu daha görünür kılmak için, v8.0.0'da kapalı bir
bloc üzerinde `add` çağırmak yakalanmamış bir istisna olarak raporlanan ve
`onError`'a iletilen bir `StateError` fırlatır.

#### ❗✨ Bloc kapalıysa `emit` `StateError` fırlatır

:::note[Ne Değişti?]

bloc v8.0.0'da, kapalı bir bloc içinde `emit` çağırmak `StateError`'a yol
açar.

:::

##### Gerekçe

Önceden, kapalı bir bloc içinde `emit` çağırmak mümkündü ve hiçbir durum
değişikliği gerçekleşmiyordu; ancak neyin yanlış gittiğine dair bir gösterge de
yoktu, bu da hata ayıklamayı zorlaştırıyordu. Bu senaryoyu daha görünür kılmak
için, v8.0.0'da kapalı bir bloc içinde `emit` çağırmak yakalanmamış bir
istisna olarak raporlanan ve `onError`'a iletilen bir `StateError` fırlatır.

```dart
class MyMockBloc extends MockBloc<MyEvent, MyState> implements MyBloc {}

  // Testler...
```

## v7.2.0

### `package:bloc`

#### ✨ Yeni `on<Event>` API'sini Tanıtma

:::note[Ne Değişti?]

bloc v7.2.0'da, `mapEventToState` `on<Event>` lehine kullanımdan kaldırıldı.
`mapEventToState` bloc v8.0.0'da kaldırılacak.

:::

##### Gerekçe

`on<Event>` API'si,
[[Öneri] mapEventToState'i Bloc'ta on\<Event\> ile değiştirme](https://github.com/felangel/bloc/issues/2526)
kapsamında tanıtıldı.
[Dart'taki bir sorun](https://github.com/dart-lang/sdk/issues/44616) nedeniyle,
iç içe asenkron generator'lar (`async*`) ile çalışılırken `state`'in
değerinin ne olacağı her zaman açık olmayabilir. Sorunun çevresinden
dolaşmanın yolları olsa da bloc kütüphanesinin temel ilkelerinden biri
öngörülebilir olmaktır. `on<Event>` API'si, kütüphaneyi olabildiğince güvenli
hale getirmek ve durum değişiklikleri söz konusu olduğunda her türlü
belirsizliği ortadan kaldırmak için oluşturuldu.

:::tip

Daha fazla bilgi için
[öneriyi tam olarak okuyun](https://github.com/felangel/bloc/issues/2526).

:::

**Özet**

`on<E>`, `E` türündeki tüm olaylar için bir olay işleyicisi kaydetmenize
imkân tanır. Varsayılan olarak, olaylar `on<E>` kullanılırken eşzamanlı
(concurrent) olarak işlenir; bu, olayları `sequential` (sıralı) olarak
işleyen `mapEventToState`'in aksine çalışır.

**v7.1.0**

```dart
abstract class CounterEvent {}
class Increment extends CounterEvent {}

class CounterBloc extends Bloc<CounterEvent, int> {
  CounterBloc() : super(0);

  @override
  Stream<int> mapEventToState(CounterEvent event) async* {
    if (event is Increment) {
      yield state + 1;
    }
  }
}
```

**v7.2.0**

```dart
abstract class CounterEvent {}
class Increment extends CounterEvent {}

class CounterBloc extends Bloc<CounterEvent, int> {
  CounterBloc() : super(0) {
    on<Increment>((event, emit) => emit(state + 1));
  }
}
```

:::note

Kayıtlı her `EventHandler` bağımsız çalışır; bu nedenle olay işleyicilerini
uygulamak istediğiniz transformer türüne göre kaydetmek önemlidir.

:::

v7.1.0'daki davranışın tamamen aynısını korumak istiyorsanız, tüm olaylar için
tek bir olay işleyicisi kaydedebilir ve bir `sequential` transformer
uygulayabilirsiniz:

```dart
import 'package:bloc/bloc.dart';
import 'package:bloc_concurrency/bloc_concurrency.dart';
```

#### ✨ Yeni `EventTransformer` API'sini Tanıtma

:::note[Ne Değişti?]

bloc v7.2.0'da, `transformEvents` `EventTransformer` API'si lehine kullanımdan
kaldırıldı. `transformEvents` bloc v8.0.0'da kaldırılacak.

:::

##### Gerekçe

`on<Event>` API'si, olay işleyicisi başına özel bir olay dönüştürücü (event
transformer) sağlama olanağının kapısını açtı. Geliştiricilerin tüm olaylar
için tek bir olay dönüştürücü belirtmek zorunda kalmak yerine, her olay
işleyicisi için gelen olay stream'ini dönüştürmesini sağlayan yeni bir
`EventTransformer` typedef'i tanıtıldı.

**Özet**

Bir `EventTransformer`, gelen olay stream'ini bir `EventMapper` (sizin olay
işleyiciniz) ile birlikte alıp yeni bir olay stream'i döndürmekten sorumludur.

```dart
typedef EventTransformer<Event> = Stream<Event> Function(Stream<Event> events, EventMapper<Event> mapper)
```

Varsayılan `EventTransformer` tüm olayları eşzamanlı olarak işler ve şuna
benzer:

```dart
EventTransformer<E> concurrent<E>() {
  return (events, mapper) => events.flatMap(mapper);
}
```

:::tip

Önerilen özel olay dönüştürücü seti için
[package:bloc_concurrency](https://pub.dev/packages/bloc_concurrency)
paketine göz atın.

:::

**v7.1.0**

```dart
@override
Stream<Transition<MyEvent, MyState>> transformEvents(events, transitionFn) {
  return events
    .debounceTime(const Duration(milliseconds: 300))
    .flatMap(transitionFn);
}
```

**v7.2.0**

```dart
/// Özel bir `EventTransformer` tanımlayın
EventTransformer<MyEvent> debounce<MyEvent>(Duration duration) {
  return (events, mapper) => events.debounceTime(duration).flatMap(mapper);
}

MyBloc() : super(MyState()) {
  /// Özel `EventTransformer`'ı `EventHandler`'a uygulayın
  on<MyEvent>(_onEvent, transformer: debounce(const Duration(milliseconds: 300)))
}
```

### `package:bloc`

#### ❗ Bloc ve Cubit, BlocBase'i miras alıyor

##### Gerekçe

Bir geliştirici olarak, bloc'lar ve cubit'ler arasındaki ilişki biraz tuhaftı.
Cubit ilk tanıtıldığında, bloc'ların temel sınıfı olarak başladı ki bu da
mantıklıydı; çünkü işlevselliğin bir alt kümesine sahipti ve bloc'lar yalnızca
Cubit'i miras alıp ek API'ler tanımlıyordu. Bunun birkaç dezavantajı vardı:

- Tüm API'lerin doğruluk için cubit kabul edecek şekilde yeniden adlandırılması
  ya da hiyerarşik olarak yanlış olmasına rağmen tutarlılık için bloc olarak
  bırakılması gerekiyordu
  ([#1708](https://github.com/felangel/bloc/issues/1708),
  [#1560](https://github.com/felangel/bloc/issues/1560)).

- Cubit'in, BlocBuilder, BlocListener vb. widget'ların karşı uygulanabileceği
  ortak bir temele sahip olması için Stream'i miras alması ve EventSink'i
  uygulaması gerekiyordu
  ([#1429](https://github.com/felangel/bloc/issues/1429)).

Daha sonra, ilişkiyi tersine çevirip bloc'u temel sınıf yapmayı denedik; bu da
yukarıdaki ilk maddeyi kısmen çözdü ama başka sorunlar getirdi:

- Cubit API'si, mapEventToState, add gibi alttaki bloc API'leri nedeniyle
  şişti ([#2228](https://github.com/felangel/bloc/issues/2228))
  - Geliştiriciler teknik olarak bu API'leri çağırıp her şeyi bozabiliyordu
- Önceki gibi cubit'in tüm Stream API'sini açığa çıkarması sorununa hâlâ
  sahiptik ([#1429](https://github.com/felangel/bloc/issues/1429))

Bu sorunları gidermek için hem `Bloc` hem de `Cubit` için `BlocBase` adında
bir temel sınıf tanıttık; böylece üst akış (upstream) bileşenleri hem bloc
hem de cubit örnekleriyle çalışabilirken `Stream` ve `EventSink` API'sinin
tamamını doğrudan açığa çıkarmamış oluyoruz.

**Özet**

```dart
  @override
  void onTransition(Bloc bloc, Transition transition) {...}
```

### `package:flutter_bloc`

#### ❗context.bloc ve context.repository, context.read ve context.watch lehine kullanımdan kaldırıldı

##### Gerekçe

`context.read`, `context.watch` ve `context.select`, birçok geliştiricinin
aşina olduğu mevcut [provider](https://pub.dev/packages/provider) API'siyle
hizalanmak ve topluluk tarafından dile getirilen sorunları gidermek için
eklendi. Kodun güvenliğini artırmak ve tutarlılığı korumak için
`context.bloc`, `build` içinde doğrudan kullanılıp kullanılmadığına bağlı
olarak `context.read` veya `context.watch` ile değiştirilebildiği için
kullanımdan kaldırıldı.

**context.watch**

`context.watch`,
[MultiBlocBuilder](https://github.com/felangel/bloc/issues/538) talebini
karşılar; çünkü birden fazla duruma göre UI render etmek için tek bir
`Builder` içinde birkaç bloc'u izleyebiliriz:

```dart
Builder(
  builder: (context) {
    final stateA = context.watch<BlocA>().state;
    final stateB = context.watch<BlocB>().state;
    final stateC = context.watch<BlocC>().state;

    // BlocA, BlocB ve BlocC'nin durumuna bağlı bir Widget döndür
  }
);
```

**context.select**

`context.select`, geliştiricilerin bir bloc durumunun bir bölümüne göre UI'ı
render etmesine/güncellemesine olanak tanır ve
[daha basit bir buildWhen](https://github.com/felangel/bloc/issues/1521)
talebini karşılar.

```dart
final name = context.select((UserBloc bloc) => bloc.state.user.name);
```

Yukarıdaki kod parçacığı, yalnızca mevcut kullanıcının adı değiştiğinde
widget'a erişip yeniden inşa etmemize olanak tanır.

**context.read**

`context.read`, `context.bloc` ile aynı görünse de bazı ince ama önemli
farklılıklar vardır. Her ikisi de bir `BuildContext` ile bir bloc'a erişmenize
olanak tanır ve yeniden inşalara yol açmaz; ancak `context.read`, doğrudan bir
`build` metodu içinde çağrılamaz. `context.bloc`'u `build` içinde kullanmanın
iki ana nedeni vardır:

1. **Bloc'un durumuna erişmek için**

```dart
@override
Widget build(BuildContext context) {
  final state = context.bloc<MyBloc>().state;
  return Text('$state');
}
```

Yukarıdaki kullanım hataya açıktır; çünkü bloc'un durumu değiştiğinde `Text`
widget'ı yeniden inşa edilmez. Bu senaryoda, ya bir `BlocBuilder` ya da
`context.watch` kullanılmalıdır.

```dart
@override
Widget build(BuildContext context) {
  final state = context.watch<MyBloc>().state;
  return Text('$state');
}
```

veya

```dart
@override
Widget build(BuildContext context) {
  return BlocBuilder<MyBloc, MyState>(
    builder: (context, state) => Text('$state'),
  );
}
```

:::note

`context.watch`'ı `build` metodunun en üstünde kullanmak, bloc durumu
değiştiğinde widget'ın tamamının yeniden inşa edilmesine yol açar. Tüm
widget'ın yeniden inşa edilmesi gerekmiyorsa, ya yeniden inşa edilmesi
gereken kısımları sarmak için `BlocBuilder` kullanın, ya yeniden inşaları
kapsamak için `context.watch` ile bir `Builder` kullanın ya da widget'ı daha
küçük widget'lara ayrıştırın.

:::

2. **Bir olay eklemek için bloc'a erişmek için**

```dart
@override
Widget build(BuildContext context) {
  final bloc = context.bloc<MyBloc>();
  return ElevatedButton(
    onPressed: () => bloc.add(MyEvent()),
    ...
  )
}
```

Yukarıdaki kullanım verimsizdir; çünkü bloc yalnızca kullanıcı `ElevatedButton`'a
dokunduğunda gerektiği halde her yeniden inşada bir bloc araması (lookup)
gerçekleşir. Bu senaryoda, bloc'a doğrudan ihtiyaç duyulan yerde
(bu durumda, `onPressed` geri çağrısında) erişmek için `context.read`
kullanmayı tercih edin.

```dart
@override
Widget build(BuildContext context) {
  return ElevatedButton(
    onPressed: () => context.read<MyBloc>().add(MyEvent()),
    ...
  )
}
```

**Özet**

**v6.0.x**

```dart
@override
Widget build(BuildContext context) {
  final bloc = context.bloc<MyBloc>();
  return ElevatedButton(
    onPressed: () => bloc.add(MyEvent()),
    ...
  )

```

**v6.1.x**

```dart

  return ElevatedButton(
    onPressed: () => context.read<MyBloc>().add(MyEvent()),
    ...
  )
}
```

?> Bir olay eklemek için bloc'a erişiyorsanız, bloc erişimini ihtiyaç duyulan
geri çağrı içinde `context.read` ile gerçekleştirin.

**v6.0.x**

```dart

}
```

**v6.1.x**

```dart

  return Text('$state');
}
```

bloc'un durumuna erişirken `context.watch`'ı kullanın.

```dart
  build: () => CounterBloc(),
  act: (bloc) => bloc.add(CounterEvent.increment),
  expect: const <int>[1],
);
```

#### ❗blocTest skip varsayılanı 0

##### Gerekçe

`bloc` ve `cubit` örnekleri artık yeni abonelikler için son durumu yaymadığı
için, `skip`'in varsayılan olarak `1` olması gerekmiyordu.

```dart

  'skip 0 olduğunda [0] yayar',

```


Bir bloc veya cubit'in başlangıç durumu şu şekilde test edilebilir:

```dart
test('başlangıç durumu doğru', () {
  expect(MyBloc().state, InitialState());
});
```

#### ❗blocTest build'i senkron yapın

##### Gerekçe

Önceden, `build` `async` yapılmıştı; böylece bloc'u test edilecek belirli bir
duruma getirmek için çeşitli hazırlıklar yapılabiliyordu. Artık gerekli değil
ve dahili olarak build ile abonelik arasındaki ek gecikmeden kaynaklanan
çeşitli sorunları da çözüyor. Bir bloc'u istenen bir duruma getirmek için
async hazırlık yapmak yerine, artık `emit`'i istenen durumla zincirleyerek
bloc'un durumunu ayarlayabiliyoruz.

**v5.x.x**

```dart
blocTest<CounterBloc, CounterEvent, int>(
  'increment eklendiğinde [2] yayar',
  build: () async {
    final bloc = CounterBloc();
    bloc.add(CounterEvent.increment);
    await bloc.take(2);
    return bloc;
  }
  act: (bloc) => bloc.add(CounterEvent.increment),
  expect: const <int>[2],
);
```

**v6.0.0**

```dart
blocTest<CounterBloc, int>(
  'increment eklendiğinde [2] yayar',
  build: () => CounterBloc()..emit(1),
  act: (bloc) => bloc.add(CounterEvent.increment),
  expect: const <int>[2],
);
```

:::note

`emit` yalnızca test için görünürdür ve testler dışında asla kullanılmamalıdır.

:::

## v5.0.0

### `package:bloc`

#### ❗initialState kaldırıldı

##### Gerekçe

Bir geliştirici olarak, bir bloc oluşturulurken `initialState`'i geçersiz
kılmak zorunda olmak iki ana sorunu doğurur:

- Bloc'un `initialState`'i dinamik olabilir ve daha sonraki bir noktada (hatta
  bloc'un dışından) referans alınabilir. Bazı açılardan bu, dahili bloc
  bilgisinin UI katmanına sızdırılması olarak görülebilir.
- Ayrıntılı (verbose) bir yaklaşımdır.

**v4.x.x**

```dart
class CounterBloc extends Bloc<CounterEvent, int> {
  @override
  int get initialState => 0;

  ...
}
```

**v5.0.0**

```dart
class CounterBloc extends Bloc<CounterEvent, int> {
  CounterBloc() : super(0);

  ...
}
```

?> Daha fazla bilgi için
[#1304](https://github.com/felangel/bloc/issues/1304) numaralı konuya bakın.

#### ❗BlocDelegate, BlocObserver olarak yeniden adlandırıldı

##### Gerekçe

`BlocDelegate` adı, sınıfın oynadığı rolün doğru bir tanımı değildi.
`BlocDelegate` adı, sınıfın aktif bir rol oynadığını ima ederken, gerçekte
`BlocDelegate`'in amaçlanan rolü, bir uygulamadaki tüm bloc'ları yalnızca
gözlemleyen pasif bir bileşen olmaktı.

:::note

`BlocObserver` içinde ideal olarak hiçbir kullanıcıya yönelik işlevsellik veya
özellik ele alınmamalıdır.

:::

**v4.x.x**

```dart
class MyBlocDelegate extends BlocDelegate {
  ...
}
```

**v5.0.0**

```dart
class MyBlocObserver extends BlocObserver {
  ...
}
```

#### ❗BlocSupervisor kaldırıldı

##### Gerekçe

`BlocSupervisor`, geliştiricilerin yalnızca özel bir `BlocDelegate`
belirlemek için bilmek ve etkileşime geçmek zorunda olduğu bir bileşen
daha demekti. `BlocObserver` değişikliğiyle birlikte, observer'ı doğrudan
bloc'un kendisi üzerinde belirlemenin geliştirici deneyimini iyileştirdiğini
hissettik.

?> Bu değişiklik ayrıca `HydratedStorage` gibi diğer bloc eklentilerini
`BlocObserver`'dan ayırmamıza da olanak tanıdı.

**v4.x.x**

```dart

```

**v5.0.0**

```dart
Bloc.observer = MyBlocObserver();
```

### `package:flutter_bloc`

#### ❗BlocBuilder condition, buildWhen olarak yeniden adlandırıldı

##### Gerekçe

`BlocBuilder` kullanırken, `builder`'ın yeniden inşa edilip edilmeyeceğini
belirlemek için önceden bir `condition` belirtebiliyorduk.

```dart
BlocBuilder<MyBloc, MyState>(
  condition: (previous, current) {
    // builder'ı çağırıp çağırmamak için true/false döndür
  },
  builder: (context, state) {...}
)
```

`condition` adı çok açıklayıcı veya açık değildi ve daha da önemlisi,
`BlocConsumer` ile etkileşime geçildiğinde API tutarsız hale geliyordu;
çünkü geliştiriciler iki koşul sağlayabiliyordu (biri `builder` ve biri
`listener` için). Sonuç olarak, `BlocConsumer` API'si bir `buildWhen` ve
`listenWhen` açığa çıkarıyordu

```dart
BlocConsumer<MyBloc, MyState>(
  listenWhen: (previous, current) {
    // listener'ı çağırıp çağırmamak için true/false döndür
  },
  listener: (context, state) {...},
  buildWhen: (previous, current) {
    // builder'ı çağırıp çağırmamak için true/false döndür
  },
  builder: (context, state) {...},
)
```

API'yi hizalamak ve daha tutarlı bir geliştirici deneyimi sağlamak için,
`condition`, `buildWhen` olarak yeniden adlandırıldı.

**v4.x.x**

```dart
BlocBuilder<MyBloc, MyState>(
  condition: (previous, current) {
    // builder'ı çağırıp çağırmamak için true/false döndür
  },
  builder: (context, state) {...}
)
```

**v5.0.0**

```dart
BlocBuilder<MyBloc, MyState>(
  buildWhen: (previous, current) {
    // builder'ı çağırıp çağırmamak için true/false döndür
  },
  builder: (context, state) {...}
)
```
